> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Produce a findings report (`docs/a11y-audit-pass1.md`) documenting fundamental accessibility issues across the ModemChecker widget.

**Architecture:** Two-layer audit — component-level checks for isolated issues (contrast, touch targets, focus rings, names, roles), then screen-level checks for contextual issues (heading hierarchy, tab order, landmarks). No code changes.

**Spec:** `docs/superpowers/specs/2026-03-15-a11y-audit-pass1-design.md`

**Key reference files:**
- Color tokens: `src/ui/theme.css` (lines 1-50 for colors, lines 173-235 for typography)
- Subframe components: `src/ui/components/`
- Page components: `src/components/`
- **Excluded:** `src/components/DevMenu.tsx` (dev-only, not shipped to production)

**Ownership tags:**
- `owned` — page components in `src/components/`, we fix directly
- `subframe-disabled` — Button, RadioCardGroup, CheckerCard, OrderCard (have `// @subframe/sync-disable` marker, we can fix)
- `subframe` — IconButton, TextField, CardButton, LinkButton, StatusItem, IconWithBackground (still synced, needs upstream fix in Subframe or sync-disable decision)

**Note:** The spec's combined "Semantic HTML & Landmarks" section is split into three more granular sections in this report: Accessible Names, Roles & Semantics, and Landmarks.

**Parallelisation:** Within Chunk 2, Tasks 3 and 4 can run in parallel. Within Chunk 3, Tasks 5 and 6 can run in parallel. Each pair writes to different report sections.

### Task 2: Audit color contrast

Read `src/ui/theme.css` to extract all color token RGB values. Then read every component listed below and identify every text-color/background-color combination. Calculate the WCAG contrast ratio for each pair using the relative luminance formula. Record all pairs in the Color Contrast section of the report.

**Reference — color tokens to extract from theme.css:**
- `--color-brand-*` (50 through 950)
- `--color-neutral-*` (50 through 950)
- `--color-error-*` (50 through 950)
- `--color-success-*` (50 through 950)
- `--color-warning-*` (50 through 950)
- `--default-font`, `--subtext-color`
- Any other color variables used in components

**Reference — typography sizes to extract from theme.css (determines normal vs large text):**
- `--text-h1` through `--text-caption` (size + weight for each)

**Reference — relative luminance formula:**
For each RGB channel: linearize (sRGB → linear), then `L = 0.2126 * R + 0.7152 * G + 0.0722 * B`. Contrast ratio = `(L1 + 0.05) / (L2 + 0.05)` where L1 is the lighter color.

**Files to read:**
- `src/ui/theme.css` (color tokens + typography tokens)
- `src/ui/components/Button.tsx`
- `src/ui/components/IconButton.tsx`
- `src/ui/components/TextField.tsx`
- `src/ui/components/RadioCardGroup.tsx`
- `src/ui/components/CardButton.tsx`
- `src/ui/components/LinkButton.tsx`
- `src/ui/components/CheckerCard.tsx`
- `src/ui/components/StatusItem.tsx`
- `src/ui/components/IconWithBackground.tsx`
- `src/ui/components/OrderCard.tsx`
- `src/components/ModemChecker.tsx` (root orchestrator — may have wrapper elements with backgrounds)
- `src/components/BaseScreen.tsx`
- `src/components/SearchInput.tsx`
- `src/components/MultipleMatches.tsx`
- `src/components/ResultCard.tsx`
- `src/components/ConditionList.tsx`
- `src/components/BottomSheet.tsx`
- `src/components/LoadingState.tsx`
- `src/components/NoMatch.tsx`
- `src/components/SearchError.tsx`
- `src/components/ErrorBoundary.tsx`
- `src/components/ModemImage.tsx`
- `src/styles/gradients.css` (gradient backgrounds)

**Steps:**

- [ ] **Step 1: Extract all color token values from theme.css**

Read `src/ui/theme.css`. Build a lookup table of CSS variable name → RGB value for every color token. Also extract typography token sizes and weights.

- [ ] **Step 2: Read all Subframe components and identify text/bg pairs**

For each Subframe component, find every Tailwind class that sets text color (`text-*`) or background color (`bg-*`). Map these to the theme tokens. Check each variant/state (default, hover, disabled, brand-primary, brand-secondary, etc.).

Note: disabled states are exempt from WCAG 1.4.3 — document them for awareness but don't flag as failures.

- [ ] **Step 3: Read all owned components and identify text/bg pairs**

Same as Step 2 but for `src/components/`. Pay special attention to:
- `BottomSheet.tsx` — gradient backgrounds (`bg-gradient-brand`, `bg-gradient-brand-compact`)
- `BaseScreen.tsx` — heading colors on neutral-50 background
- `LoadingState.tsx` — text on whatever background the sheet provides
- `ErrorBoundary.tsx` — fallback UI colors

- [ ] **Step 4: Calculate contrast ratios and write findings**

For each text/bg pair, calculate the WCAG contrast ratio. Determine if the text qualifies as "large text" (≥24px, or ≥18.66px and bold ≥700). Apply the appropriate threshold (3:1 for large, 4.5:1 for normal). Write all findings to the Color Contrast section.

Finding format:
```
- [ownership] [severity] **ComponentName** (variant): `text-color` on `bg-color` — X.X:1 (text size, weight) ✅/❌
```

Tag trivially fixable items (e.g. swap to a darker shade) as `quick-win`.

### Task 4: Audit focus rings

Read every interactive component and check for `:focus-visible` styles. Assess consistency of ring appearance (color, width, offset) across the component set.

**What to look for in each component:**
- `focus-visible:ring-*` — explicit focus ring ✅
- `focus-visible:outline-*` — explicit focus outline ✅
- `focus-within:ring-*` — container focus ring (check if the actual interactive child also has one)
- No focus-visible styles at all — relies on browser default ❌
- `outline-none` without a replacement ring — focus indicator removed ❌

**Components to check (same interactive list as Task 3):**
- Button (`src/ui/components/Button.tsx`)
- IconButton (`src/ui/components/IconButton.tsx`)
- TextField (`src/ui/components/TextField.tsx`)

Also check page components that add their own interactive wrappers:
- MultipleMatches (`src/components/MultipleMatches.tsx`) — CardButton with role="button"
- SearchInput (`src/components/SearchInput.tsx`)

**Steps:**

- [ ] **Step 1: Read all interactive components and catalogue focus styles**

For each component, search for `focus-visible`, `focus-within`, `focus:`, `outline`, and `ring` in Tailwind classes. Record exactly what styles are applied.

- [ ] **Step 2: Assess consistency**

Compare ring color, width, and offset across all components. The project uses `ring-brand-600` and `ring-offset-2` on CardButton and RadioCard. Check if other components match or diverge.

- [ ] **Step 3: Write findings to Focus Rings section**

## Chunk 3: Accessible Names + Roles & Semantics

### Task 5: Audit accessible names

Check every interactive element has a programmatically determinable accessible name.

**What to check per element type:**
- **Buttons with text:** visible text content is the name ✅
- **Icon-only buttons (IconButton):** needs `aria-label` or visually hidden text
- **Inputs:** needs associated `<label>` (via wrapping or `htmlFor`)
- **Images:** needs `alt` attribute (decorative images: `alt=""`)
- **Links/LinkButtons:** visible text content is the name ✅

**Components to check:**
- All Subframe interactive components (Button, IconButton, TextField, RadioCardGroup, CardButton, LinkButton, OrderCard)

- [ ] **Step 1: Read all components and check accessible names**

- IconButton instances in SearchInput (close button) and MultipleMatches (close/back buttons) — do they have `aria-label`?
- TextField.Input in SearchInput — is it associated with a `<label>`?
- CardButton images in MultipleMatches — do they have `alt`?
- ModemImage — does it always provide `alt`?

- [ ] **Step 2: Write findings to Accessible Names section**

### Task 6: Audit roles & semantics

Check that clickable elements use correct HTML elements or ARIA roles, and that existing ARIA attributes are correct.

**What to check:**
- Every clickable element: is it a `<button>`, `<a>`, `<input>`, or does it have `role="button"` + keyboard handler (Enter/Space)?
- Every `<div>` or `<span>` with `onClick`: must have `role`, `tabIndex`, and keyboard handler
- Existing ARIA attributes: check they are correct (e.g. `aria-modal="true"` on BottomSheet, `role="status"` on LoadingState, `aria-describedby={undefined}` on BottomSheet)
- `<img>` elements: have `alt`
- `<form>` elements: have accessible submit mechanism

**Components to check:**
- CardButton (`src/ui/components/CardButton.tsx`) — uses `<div>` with `role="button"`, verify keyboard handler
- OrderCard (`src/ui/components/OrderCard.tsx`) — has `onClick` and `cursor-pointer`, check if it's a native button or needs role
- ModemChecker (`src/components/ModemChecker.tsx`) — root orchestrator, check for any wrapper elements with click handlers
- BottomSheet (`src/components/BottomSheet.tsx`) — check `aria-modal`, `aria-describedby`, Dialog.Title (sr-only)
- LoadingState (`src/components/LoadingState.tsx`) — check `role="status"` is present and correct
- All other owned components — spot-check for `<div onClick>` anti-patterns

**Steps:**

- [ ] **Step 1: Read all components and check semantic correctness**
